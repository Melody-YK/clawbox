# ClawBox 项目补充说明（2026-05-15）

本文是对现有《总结.docx》与《clawbox-现场修复记录-中文版》之外的补充说明。

写作边界：
- 不重复系统 bring-up、Debian/NetworkManager/SSH 检查、依赖安装、服务首次拉起这类内容。
- 不重复“设备已联网、服务已启动、setup 主链路已走通、微信通道已打通”这类阶段性结论。
- 重点补充半路接手后实际做过的“代码收口、工程化修补、验证补全、交付风险控制”。

---

## 一、接手后真正补上的，不只是功能，还有“可维护性”

这个项目最初的问题，不只是若干接口没写完，而是“能跑”和“能接手、能复现、能回滚、能继续维护”之间还有明显断层。接手后补的工作，核心是把它从“某台样机上勉强跑通”往“可以被别人继续接住并稳定迭代”的方向推。

主要体现在四件事上：

1. 把产品入口和运行引擎重新分层  
   原先项目容易让人把 ClawBox 与 OpenClaw 混成同一个 Web。后续代码已经明确把 ClawBox 定位为设备 setup / 管理壳层，把 OpenClaw 定位为后端能力与 gateway，不再默认让终端用户直接落入 OpenClaw 自带 Web。

2. 把“提交配置成功”改成“真实状态成功”  
   这在 WiFi、AI 配置、微信配置三个地方都做了同一类收口：不能只看接口调用是否返回 200，而要看设备是否真的进入了目标状态。

3. 把临时人工处理的流程变成代码内约束  
   包括 WiFi 成功判定、setup 完成条件、微信前置条件、gateway restart 成功条件、二维码超时和日志诊断等，尽量减少“靠人记住下一步该怎么点/怎么补”的部分。

4. 把交接风险显性化  
   比如运行目录与源码目录分离、保留 `data/`、不建议直接在运行目录拉代码、AP 服务不作为远程维护默认重启项、Windows 传输后补脚本可执行权限，这些都不是单纯“修 bug”，而是在给后续接手者降风险。

## 二、代码层真正完成的收口项

### 1. WiFi 链路已经从“表面成功”改成“真实联网成功”

这一块是比较关键的质量提升，涉及的不是一个接口，而是一整条状态链：

- `src/app/setup-api/wifi/connect/route.ts`
- `src/lib/network.ts`
- `src/lib/setup-status.ts`
- `src/app/setup-api/setup/status/route.ts`
- `src/app/setup-api/setup/complete/route.ts`
- `src/app/setup-api/wifi/status/route.ts`
- `src/components/WifiStep.tsx`
- `src/components/SetupWizard.tsx`
- `src/components/DoneStep.tsx`

实际完成的收口包括：

- WiFi 提交后不再立刻视为成功。
- 只有在设备真实拿到 DHCP IPv4 后，才会写入 `wifi_configured=true`。
- 增加了 WiFi 过程态字段：
  - `wifi_connecting`
  - `wifi_target_ssid`
  - `wifi_last_error`
  - `wifi_last_attempt_at`
- `setup/status` 已经具备“真实状态自愈”能力：
  - 即使配置文件还没写成功，只要运行时已经 `connected + ipv4`，接口也能返回正确状态并补写配置。
- `Finish Setup` 不再由 WiFi 页偷偷代调用，setup 完成条件被收回到统一状态判断。

这意味着，之前文档里提到的“设备明明已经连上 WiFi，但 `wifi_configured` 没变”这类问题，已经不再只是现场人工发现的问题，而是有了代码层的修复路径。

### 2. mDNS / 主机名 / OLED 这条链已经成体系了

这部分不是单个字段补丁，而是设备访问方式的整体收口，涉及：

- `src/lib/device-identity.ts`
- `src/lib/network.ts`
- `src/lib/system-info.ts`
- `install.sh`
- `config/clawbox-http.service.xml`
- `scripts/start-ap.sh`
- `scripts/oled-display.py`

已经落实的实现点包括：

- 设备默认主机名不再是容易冲突的固定值，而是稳定派生的 `clawbox-<suffix>`。
- 默认访问入口统一到：
  - `http://clawbox-<suffix>.local/`
- `clawbox.home.arpa` 被收口为“可选本地 DNS 别名”，不是默认发现机制。
- 增加了 `mdnsReady` 设备侧判定，区分：
  - 设备没正确发布
  - 客户端不支持 `.local`
- OLED 展示逻辑也同步调整为：
  - 优先显示 `.local`
  - 拿不到 IPv4 时显示 `Waiting for DHCP`
  - 有 IPv4 时显示 `.local + IP`

这部分价值在于：访问方式不再只停留在口头方案，而是已经落到了“设备名生成、服务发布、状态回传、屏幕显示、安装脚本”五个层面。

### 3. 产品入口已经和 OpenClaw Web 做了切割

这里是一个很重要但容易被忽略的改动：

- `src/app/route.ts`
- `src/app/[...gateway]/route.ts`

当前代码口径已经明确：

- `/` 默认重定向到 `/setup`
- `[...,gateway]` 默认不作为最终用户入口
- 只有显式设置 `ENABLE_GATEWAY_WEB_DEBUG=1` 才允许通过它进入 gateway Web

也就是说，这个项目已经不再把“OpenClaw 自带 Web 页面”当作最终产品默认首页，而是回到 ClawBox 自己的面板和引导逻辑上。

这点和甲方后续“主要通过微信通道使用，不依赖 OpenClaw 原生 Web”这个方向是一致的。

### 4. AI / 微信两段式引导，已经从页面文案变成后端约束

涉及文件：

- `src/components/DoneStep.tsx`
- `src/app/setup-api/ai-models/configure/route.ts`
- `src/app/setup-api/wechat/configure/route.ts`
- `src/lib/openclaw-config.ts`

已经落下去的约束包括：

- AI 区优先于微信区。
- AI 未完成前，微信区可见但不可保存。
- `POST /setup-api/wechat/configure`
  - AI 未完成时返回 `409`
  - gateway restart 失败时返回 `502`
- `POST /setup-api/setup/complete`
  - 只有 `wifi_configured && ai_model_configured` 才允许成功
- `POST /setup-api/ai-models/configure`
  - 只有凭据写入成功且 gateway restart 成功，才写 `ai_model_configured=true`

这意味着：

- “先配 AI，再配微信”不再只是页面提醒。
- 即使有人绕过前端直接调接口，也不会提前写成成功状态。

### 5. WeChat 二维码登录已经从“能用”改成“可诊断”

除了现有现场记录里写到的超时修复，代码里还补了更有交接价值的东西：

- `src/app/setup-api/wechat/qrcode/route.ts`
- `src/app/setup-api/wechat/login-status/route.ts`
- `src/components/DoneStep.tsx`
- `src/lib/openclaw-config.ts`

它的提升不只是“把二维码搞出来”，而是：

- 登录输出会先做 ANSI 清洗，再提取真正可用的微信扫码 URL。
- 冷启动慢时，错误里保留 output tail，后续排查不会再是“只知道超时，不知道卡在哪”。
- 增加了独立的 `login-status` 状态接口，用真实账户文件和连接状态去判定是否登录成功。
- 前端 WeChat 完成态不再只依赖 `enabled` 这种弱状态，而是回到 `connected`。

这类改动的意义在于，WeChat 链路从“现场靠人盯日志判断”逐步变成“前后端都能解释当前状态”。

## 四、半路接手后补上的工程化能力

这部分通常不会出现在功能汇报里，但对后续交接非常重要。

### 1. 运行目录和源码目录的边界被明确了

已形成的建议和操作口径是：

- 源码目录：`/home/pi/clawbox-src`
- 运行目录：`/home/clawbox/clawbox`

其意义在于：

- 不直接在运行目录里拉代码
- 不混淆运行时数据与源码
- 可通过 `rsync` 做可控覆盖
- 部署时保住 `data/`

这比“把改好的代码再想办法拷进去”高了一个层次，属于交付流程层面的修复。

### 2. 远程维护的高风险动作已经被识别出来

现场日志之外，真正值得沉淀的经验是：

- 重启 `clawbox-ap` 可能导致设备切回热点，从而把当前局域网 SSH 路径打断。
- Windows 直接传输压缩包或文件时，Shell 脚本可执行位可能丢失，导致 systemd 报 `203/EXEC`。
- `.next` 产物如果中断重建，可能留下零字节 manifest，进而引发 setup 服务重启循环。

也就是说，当前项目已经暴露出几个“不是功能 bug，而是部署/维护级故障源”的点，而且这些点已经被识别并有了应对策略。

### 3. 现场日志和当前工作区之间还存在一个版本管理提醒

值得单独记录的是：

- 现场修复日志里提到曾建立本地 Git 基线和检查点提交。
- 但当前这份 `D:\workspace\clawbox-main\clawbox-main` 目录，今天再次检查时并没有 `.git`。

这说明至少有一种可能：

- 曾用于修复的工作副本，与当前交接目录不是同一份目录；
- 或者 Git 基线没有保留在当前交接副本里。

这不影响当前代码能读、能跑、能测，但它会影响后续的：

- 版本追踪
- 回滚定位
- 谁改了什么的审计能力

这是一条管理层面需要知道的真实情况。

## 五、当前还没完全闭环，但已经暴露出来的分叉点

### 1. DeepSeek 运行时与本地 Web provider 列表可能不一致

现场日志里已经记录：

- 设备运行时已切到 `deepseek/deepseek-v4-flash`

但当前本地 Web / AI 配置代码里，显式 provider 列表仍主要是：

- `anthropic`
- `openai`
- `google`
- `openrouter`

这意味着一个真实分叉风险：

- 设备现场运行态
- 本地 Web 配置壳层

二者现在不一定完全一致。

短期内它不一定阻塞使用，但后续如果继续让不同人接手，很容易出现：

- Web 上看见的是一套 provider 选项
- 设备运行的却是另一套模型和认证配置

这需要后续专门做一次统一。

### 2. 当前“代码很完整”不等于“交付版已经最终定稿”

现在更准确的状态是：

- 本地代码逻辑已比较完整
- 自动化验证已成体系
- 树莓派现场主链路已被多轮修通
- 但正式量产交付前，仍然需要再确认：
  - 当前树莓派运行目录是否已完整同步到最新本地代码
  - provider 口径是否统一
  - `.local` 在不同客户端上的体验文案是否还需继续收口
  - 现场 SOP 是否需要进一步脚本化

所以当前项目最准确的评价不是“已完工”，而是：

> 已从半成品推进到可持续联调版本，主链路具备继续收口和交付验证的条件。

## 八、2026-05-18 现场补充（新增，不与前文重复）

### 1) WiFi 切网后的“可访问地址生成时机”再次收口

本轮新增目标是：

- 不在服务启动时预设访问地址；
- 改为用户输入正确 WiFi 密码后，设备从 AP 模式切到 WiFi client 模式并拿到 DHCP IPv4 后，再生成“可直接访问地址”；
- 前端给出可点击跳转入口，降低用户手动输入成本。

已落地的关键变更：

- `src/lib/network.ts`
  - `WifiRuntimeState` 增加：`accessUrl` / `ipv4Url` / `mdnsHost`。
- `src/lib/setup-status.ts`
  - `SetupStatus` 增加：`wifi_ready_url` / `wifi_ipv4` / `wifi_ipv4_url`。
  - 运行时已连上且拿到 IPv4 时，会同步写回：`wifi_access_url`、`wifi_ipv4`。
  - 回退热点或 pending 超时时，会清理上述字段，避免残留脏状态。
- `src/app/setup-api/wifi/connect/route.ts`
  - WiFi 连接成功后持久化 `wifi_access_url`、`wifi_ipv4`。
  - 返回文案从“提示用户自行猜地址”改为“等待后给出可跳转地址”。
- `src/components/SetupWizard.tsx`
  - 轮询 `setup/status` 时识别新增字段，连接成功后进入“可跳转”提示态。
- `src/components/WifiStep.tsx`
  - 成功态新增 `Open Device` 按钮。
  - 展示完整地址与 IPv4 fallback（含可复制/可点击用途）。

### 2) 行为解释：刷新后从 WiFi 页进入大模型/微信页是当前逻辑预期

现场现象：

- 仍在旧版固定 `.local` 路径下；
- 刷新后切到大模型/微信配置区。

原因：

- 这是 Setup 向导内部 step 切换（不是路由跳到别的站点）；
- 当 `wifi_configured=true` 且 `hotspot_active=false`，向导会自动进入下一步。

### 3) 本轮验证结果（真机）

- `npm run build` 通过；
- `systemctl restart clawbox-setup` 后服务状态 `active`；
- `curl http://127.0.0.1/setup-api/setup/status` 已返回新增字段（示例）：
  - `wifi_ready_url: "http://192.168.31.55/"`
  - `wifi_ipv4: "192.168.31.55"`
  - `wifi_ipv4_url: "http://192.168.31.55/"`
